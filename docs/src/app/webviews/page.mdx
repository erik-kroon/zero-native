# Overlay WebViews

Overlay WebViews let an app place an isolated native WebView above an existing window. They are useful for previews, auth flows, embedded docs, and other content that should not receive the app bridge.

## JavaScript API

Enable the built-in WebView helpers from an allowed origin:

```zig
const app_permissions = [_][]const u8{zero_native.security.permission_window};

.security = .{
    .permissions = &app_permissions,
    .navigation = .{ .allowed_origins = &.{ "zero://app", "https://example.com" } },
},
.js_window_api = true,
```

Then create and manage an overlay from JavaScript:

```javascript
const preview = await window.zero.webviews.create({
  label: "preview",
  url: "https://example.com",
  frame: { x: 24, y: 24, width: 480, height: 320 },
});

await preview.setFrame({ x: 32, y: 32, width: 640, height: 420 });
await preview.navigate("https://example.com/docs");
await preview.setZoom(1.25);
await preview.close();
```

Coordinates are logical content coordinates relative to the parent window: `x: 0, y: 0` is the top-left corner of the window content area.

`windowId` defaults to the window that calls the command. If you pass it explicitly, it must match the calling window; one window cannot manage another window's overlays.

You can also enable the same helpers with an explicit `builtin_bridge` policy instead of `js_window_api`. In that mode, list every overlay command your app calls: `zero-native.overlay.create`, `zero-native.overlay.setFrame`, `zero-native.overlay.navigate`, `zero-native.overlay.setZoom`, and `zero-native.overlay.close`.

## Lifecycle

Overlay labels are unique per parent window. Creating a second overlay with the same label fails; close the existing overlay first if you want to replace it.

When a parent window closes, its overlays are closed with it. Overlay pages do not receive `window.zero` by default, so untrusted content cannot call native bridge commands through the app bridge.

Overlay slots are capped globally per runtime, not per window. The default cap is 16 overlays.

Overlay URLs are checked against the runtime navigation policy before native views are created or navigated. Add the target origin to `security.navigation.allowed_origins` before loading it.

## Backend Support

System WebView overlays are implemented on macOS and Linux, including page zoom. Windows overlays require WebView2; if WebView2 is unavailable, creating an overlay fails instead of showing a placeholder. Chromium/CEF overlay support is available on macOS, including page zoom. The Linux CEF host currently fails overlay creation until a real native CEF overlay implementation is wired in.
